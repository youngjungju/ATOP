# PLAN.md
# 1-Day Build Plan (Cursor-Oriented)

## 0. 프로젝트 원칙

- 빠르게 만들되, 위험한 기능은 만들지 않는다
- AI는 한 곳(Vision 관찰 요약)에만 사용한다
- 복용 약 정보는 기록만 하며, 판단하지 않는다
- "돌아가는 것"이 최우선이다

---

## 1. Phase 1 — 구조 세팅 (30분)

### 작업 내용
- Next.js App Router 초기화
- 기본 라우트 생성

### 생성 파일
- app/page.tsx (Landing)
- app/upload/page.tsx
- app/report/[id]/page.tsx

---

## 2. Phase 2 — 이미지 업로드 & 저장 (1시간)

### 작업 내용
- Supabase Storage bucket 생성
- 이미지 업로드 API 구현
- 업로드 성공 시 public URL 확보

### 생성 파일
- lib/supabase.ts
- app/api/upload/route.ts

### 체크포인트
- 이미지를 실제로 업로드 가능해야 함
- 브라우저에서 이미지 URL 접근 가능

---

## 3. Phase 3 — Vision 관찰 요약 API (1시간)

### 작업 내용
- OpenAI Responses API 연결
- Vision 입력 + 제한 프롬프트 구성
- 시각적 특징 관찰 요약 텍스트 반환

### 생성 파일
- lib/openai.ts
- app/api/vision/route.ts

### 체크포인트
- 이미지 URL을 넣으면 요약 텍스트가 반환됨
- 진단/치료/약 추천 표현이 없는지 확인

---

## 4. Phase 4 — 약 복용 기록 UX 추가 (30분)

### 작업 내용
- 업로드 페이지에 약 정보 입력 UI 추가
- 약 이름 입력 필드
- 복용 시간 선택 (아침/저녁)
- 타입 선택 (바르는 약 / 먹는 약)
- Vision API 요청 시 함께 전달

### 체크포인트
- 약 정보가 리포트 페이지에 그대로 표시됨
- AI가 약에 대해 판단하지 않음

---

## 5. Phase 5 — 업로드 → 리포트 플로우 연결 (1시간)

### 작업 내용
- 업로드 페이지에서:
  - 파일 선택
  - 이미지 업로드 API 호출
  - Vision API 호출
  - 약 정보와 함께 결과 전달
- report 페이지로 이동

### 구현 방식
- MVP에서는 DB 저장 없이 URL 파라미터 또는 인메모리 전달 허용

### 체크포인트
- 버튼 한 번으로 리포트 페이지까지 이동

---

## 6. Phase 6 — 리포트 UI & 문구 정리 (45분)

### 작업 내용
- 리포트 카드 UI 구성
- 약 정보 섹션 시각적으로 구분
- 고지 문구 명확화
- 발표용 문장 정리

### 필수 문구
- "의학적 진단이 아닙니다"
- "의사와의 소통을 돕는 관찰 리포트입니다"

---

## 7. Phase 7 — 랜딩 페이지 정리 (30분)

### 작업 내용
- 서비스 목적 명확화
- 병원 커뮤니케이션 문제 정의
- CTA 버튼 배치
- 의료 오해 방지 문구 포함

---

## 8. Phase 8 — 데모 시나리오 점검 (30분)

### 데모 플로우
1. 랜딩 → 서비스 목적 설명
2. 사진 업로드
3. 약 복용 정보 입력
4. AI 관찰 요약 생성
5. 리포트 화면 제시
6. 병원 커뮤니케이션 시나리오 설명

---

## 9. 최종 체크리스트

- [ ] 실제 이미지 업로드 가능
- [ ] Vision 관찰 요약 결과 출력
- [ ] 약 정보 입력 및 리포트 반영
- [ ] 의료 진단 아님 고지 확인
- [ ] 1분 내 데모 가능

---

## 10. 발표용 핵심 문장

> 이 서비스는 AI에게 진단을 맡기지 않습니다.  
> 대신, 사진과 복용 기록을 정리해  
> **의사에게 설명하기 쉬운 언어로 바꿔주는 도구**입니다.
